3. Course Project Milestones Comprehensive full-stack systems engineering architectures showcasing incremental software maturation.

Project 1: Navigational Flow & Feature Blueprint (Lab 4 Extension)

🔴 Problem Statement: Hikers often get confused when an app has messy layouts and disorganized trail info. Without a clear interface blueprint and proper navigation planning, switching between maps, weather screens, and user feeds can feel clunky, lose data, or break the app's back button behavior.

✨ Working Features Added:

  • 7-Screen UI Architecture: Created the complete visual flow of the app using standard Material 3 Scaffold layouts, custom TopBars, and interactive buttons.
  • Jetpack NavHost Routing: Set up a centralized navigation graph to manage clean, error-free transitions across all 7 main screens.
  • Data Passing Between Screens: Implemented safe argument bundles to pass trail info smoothly when a user clicks a item on the home list to open the detail view.
  • Mock Data Setup: Built hardcoded object models and list arrays to act as a placeholder framework before linking up the live database later.

🧠 Lessons Learned: I learned how to properly structure multi-screen apps by separating layout designs from state logic using State Hoisting. Managing screen destinations within a single navigation graph also taught me how to keep the app's backstack clean, ensuring users don't face lag or broken navigation paths when pressing the back button.

Project 2: Sensor-Aware Full-Stack Architecture (Lab 5 Extension)

🔴 Problem Statement: Most hiking apps completely lose functionality or crash inside deep forest terrains due to zero network coverage. Without offline data caching and live environmental alerts, hikers are exposed to navigational risks, and their tracking logs can easily be lost.

✨ Working Features Added:

  • API Data Fetching: Used Retrofit to asynchronously pull dynamic trail listings from a MockAPI endpoint into the app.
  • Live GPS & Mapping: Integrated the phone's native GPS sensor with Osmdroid to display the hiker's real-time position on an interactive map.
  • Real-Time Weather Alerts: Connected to the Open-Meteo REST API to fetch up-to-date weather conditions at specific trail locations for safety.
  • Hybrid Offline-Online Caching: Implemented Room Database to save reviews locally when offline, which automatically syncs to Firebase Firestore once internet access is restored.

🧠 Lessons Learned: I learned how to manage multiple data flows simultaneously using Kotlin Coroutines so that the UI stays smooth and doesn't freeze while fetching background GPS data. I also gained a solid understanding of combining local storage (Room) with cloud storage (Firebase) to keep user data reliable under weak network conditions.

4. Learning Journey Reflection A holistic retrospection of shifts in personal technical capabilities and code maturity.

🚀 Evolution (From Lab 1 to Project 2)

My engineering confidence shifted fundamentally. I began in Lab 1 by creating hardcoded UI assets. By the time I reached Project 2, I was independently architecting reactive Android software capable of mapping real-time geospatial configurations and caching multi-tier datasets.

⚠️ Technical Challenges Overcome

Handling thread-safety concurrency issues while mapping dynamic hardware location updates onto localized database streams represented a critical learning roadblock. I mitigated this by wrapping continuous GPS emission streams inside localized async background Coroutines, avoiding primary main thread freezes.

🌳 SDG Structural Impact

Developing this eco-hiking solution bridges digital infrastructure with active environmental safety frameworks (SDG 15). Providing data-driven mapping variables keeps community hikers informed, reducing accidents and promoting sustainable trail habits.
